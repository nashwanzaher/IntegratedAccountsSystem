```
│  - SelectData()  (FUNCTION-first fallback)  │

└─────────────────────────────────────────────┘
```

**Rule of thumb:** a layer may only call the layer directly below it. UI must
not write SQL. BL must not open a `NpgsqlConnection`. DAL must not contain
business rules.

## 7. PostgreSQL Function-vs-Procedure Pattern (critical!)

PostgreSQL functions and procedures are **NOT** interchangeable:

| Type | Use `CALL` | Use `SELECT * FROM fn(...)` | Return type |
|------|-----------|------------------------------|-------------|
| **FUNCTION** | ❌ (error 42883) | ✅ | scalar / table / row |
| **PROCEDURE** | ✅ (CommandType.StoredProcedure) | ❌ | OUT params / void |

The DAL's `clsCN.SelectData` and `DbContext.SelectData` already implement a
**try-PROCEDURE-then-fallback-to-FUNCTION** strategy. When you call them, just
pass the object name; the wrapper handles dispatching.

**Exception codes to know:**
- `42883` — function/procedure does not exist
- `42809` — wrong object kind (e.g., CALL on a function)
- `42703` — undefined column
- `23505` — unique violation
- `23503` — FK violation

## 9. Agent Behavior Guidelines

### 9.1 When working with C# code

1. Always use the DAL (`DbContext` for new code, `clsCN` for legacy code).
2. Wrap DB calls in `using var ctx = new DbContext(); ctx.Open();`.
3. For write operations, use `BeginTransaction()` / `CommitTransaction()`.
4. Use the existing `cloneParameters` helper when reusing `NpgsqlParameter[]`.
5. Prefer `CommandType.Text` + `SELECT * FROM fn(...)` for new functions.
6. Run `dotnet build` after every change; never leave warnings behind.

### 9.2 When working with SQL / the database

1. Tables and columns stay in PascalCase; variables and JSON keys in
   snake_case.
2. Every `CREATE FUNCTION` must declare `LANGUAGE plpgsql` and a `STABLE |
   VOLATILE | IMMUTABLE` volatility marker.
3. Test every new function/procedure in `psql` **before** calling from C#.
4. After any schema change, re-run `DbTest` to verify the 46 workflows.
5. Document new objects in the matching `IntegratedAccSys_*.sql` file.

### 9.3 When working with WinForms (PL)

1. Forms live in `src/IntegratedAccSys.PL/{Module}/frm*.cs`.
2. Apply privileges through `PrivilegeApplier` — never bypass the matrix.
3. Use `RDLC` reports from `Reports/`; do not hard-code SQL in reports.
4. All UI strings are bilingual (Arabic primary, English secondary).

### 9.4 When debugging

1. Check `cleanup-audit.log`, `mv-baseline.log`, `seed*.log` in the root.
2. Run `scripts/inventory-live-db.ps1` for a current DB snapshot.
3. Use `psql -c "EXPLAIN ANALYZE …"` for slow queries.
4. For connection issues, verify `DalSettings` and the env vars
   (`PGHOST`, `PGPORT`, `PGUSER`, `PGPASSWORD`).

### 9.5 Forbidden actions

- ❌ Never commit secrets, passwords, or `MINIMAX_API_KEY` values.
- ❌ Never drop a live table without an explicit backup.
- ❌ Never call `CALL getUserForLogin(...)` — it's a FUNCTION, use `SELECT`.
- ❌ Never write SQL inside WinForms code-behind.
- ❌ Never edit files inside `bin/` or `obj/`.
